# Museum of AI Art

A fully autonomous, self-curating art gallery. Every day, a real historical
event becomes a single generated oil painting, hung permanently on a static
website. No backend. No database. The git repository *is* the database.
GitHub Actions + GitHub Pages run and host the whole thing for free
(except a few cents of API cost per day).

The collection grows without human intervention. We do not rotate, we do not
retire, and we do not edit.

---

## How a painting gets made

```
 06:00 UTC, every day
        │
        ▼
 ┌────────────────────┐    ┌─────────────────────┐
 │ Wikipedia          │    │ OSS LLM via         │
 │ On-This-Day feed   │──▶ │ OpenRouter          │
 │ (free, no key)     │    │ (title, medium,      │
                      │    │  statement, prompt)  │
 └────────────────────┘    └──────────┬──────────┘
                                      │
                                      ▼
                            ┌─────────────────────┐
                            │ WaveSpeed AI Flux   │
                            │ (1024×768 oil)      │
                            └──────────┬──────────┘
                                      │
                                      ▼
                ┌───────────────────────────────────┐
                │  gallery/exhibits/YYYY-MM-DD/      │
                │    ├── painting.jpg                │
                │    └── meta.json                   │
                │  gallery/exhibit.json  (index)     │
                └───────────────────────────────────┘
                                      │
                                      ▼
                          git commit + git push
                                      │
                                      ▼
                       GitHub Pages rebuilds (~30s)
```

The script that does all of this is `scripts/curator.py` — about 350 lines,
no third-party services beyond the two APIs, and idempotent per calendar day
(re-runs are no-ops).

## Running your own

1. **Fork this repository.**
2. **Add two secrets** under *Settings → Secrets and variables → Actions*:
   - `OPENROUTER_API_KEY` — from [openrouter.ai/keys](https://openrouter.ai/keys).
     Funds the curator LLM (the default free OSS tier is also supported).
   - `WAVESPEED_AI_API_KEY` — from [wavespeed.ai](https://wavespeed.ai). Funds
     image generation on WaveSpeed's Flux 2 Dev endpoint.
3. **Enable GitHub Pages** under *Settings → Pages*:
   - Source: *Deploy from a branch*
   - Branch: `main`, folder: `/gallery`
4. **Wait for the cron**, or trigger the *Backfill* workflow from the Actions
   tab to seed the last 7 days of paintings immediately.

That's it. The site is live at `https://<your-username>.github.io/<repo>/`.

### Local development

```bash
pip install -r requirements.txt
export OPENROUTER_API_KEY=...
export WAVESPEED_AI_API_KEY=...
export MUSEUM_DRY_RUN=1         # skip WaveSpeed, write a 1×1 stub painting
export MUSEUM_DATE=2024-07-04   # optional: backfill a specific day
python scripts/curator.py
```

`MUSEUM_DRY_RUN=1` is useful for testing the pipeline end-to-end without
incurring image-generation cost. Set `MUSEUM_OPENROUTER_MODEL` and
`MUSEUM_WAVESPEED_MODEL` to swap models without editing code.

## Cost

| Component                | Cost per day                |
|--------------------------|-----------------------------|
| Wikipedia REST API       | Free                        |
| OpenRouter OSS curator   | Free (default `:free` model) or ≈ $0.0001 |
| WaveSpeed AI Flux 2 dev  | ≈ $0.012 per image          |
| GitHub Actions minutes   | Free (well under 1 min/day) |
| GitHub Pages bandwidth   | Free                        |
| **Total**                | **≈ $0.012 / day**          |

A month of operation is roughly the price of a sandwich. Most of that is
image generation, and it has dropped ~4× since switching from fal.ai to
WaveSpeed.

## Architecture notes

- **The git repo is the database.** Every painting is a directory of two
  files: `painting.jpg` and `meta.json`. `gallery/exhibit.json` is a
  build-time index, regenerated by `curator.py` at the end of each run.
- **Pages serves `gallery/` as the site root.** The front-end (`index.html`,
  `styles.css`, `app.js`) is plain ES2015 with no build step.
- **The front-end reads `exhibit.json` at runtime** to render the wall. For
  the featured painting it additionally fetches the per-day `meta.json` to
  show the full artist statement. Everything is cacheable static HTTP.
- **Concurrency safety.** Multiple workflow runs (e.g. the cron and a manual
  backfill firing at the same time) are serialised by the
  `concurrency: museum-curator` group in `daily.yml`. The per-day idempotency
  in `curator.py` is the second line of defence.
- **Determinism per day.** The random event picker is seeded with
  `museum-YYYY-MM-DD`, so a backfilled day will always pick the same event
  even if the script is re-run.

## File map

```
.github/workflows/
  daily.yml        # cron at 06:00 UTC + push trigger
  backfill.yml     # manual seed of past N days
scripts/
  curator.py       # the only script that matters
gallery/
  index.html       # the museum site
  styles.css
  app.js
  exhibit.json     # generated; do not edit by hand
  exhibits/
    YYYY-MM-DD/
      painting.jpg
      meta.json
```
